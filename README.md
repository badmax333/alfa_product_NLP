## 1. Обзор проекта

**Проблема:** Бизнес-клиенты (ЮЛ, ИП, КФХ) открывают расчётный счёт в банке, но не активируют банковские продукты — остаются в статусе «спящих». Традиционный онбординг использует единый скрипт для всех типов клиентов, игнорируя отраслевую специфику, размер бизнеса и реальные потребности.

**Решение:** Трёхступенчатая система персонализированного онбординга на базе ML + LLM. На каждом этапе жизненного цикла клиента система применяет релевантную модель данных и генерирует персонализированный аргумент для активации нужного продукта.

**Цель POC:** Подтвердить гипотезу о том, что персонализированный аргумент, сгенерированный на основе профиля клиента, повышает конверсию в активацию якорного продукта по сравнению с базовым скриптом.

```
Клиент открыл счёт
        │
        ▼
┌───────────────────┐    День 0–4     ┌─────────────────────────────┐
│   СТУПЕНЬ 1       │ ──────────────► │  Статический классификатор  │
│  Холодный старт   │                 │  (CatBoost, 8 сегментов)    │
│  Регистр. данные  │                 │  → Якорный продукт + скрипт │
└───────────────────┘                 └─────────────────────────────┘
        │
        ▼ (накопление транзакций 4–30 дней)
┌───────────────────┐    День 4–30    ┌─────────────────────────────┐
│   СТУПЕНЬ 2       │ ──────────────► │  Модели склонности (AUC>0.65│
│  Транзакц. история│                 │  SHAP → LLM-аргумент        │
│  30+ событий      │                 │  (батч-генерация, 1/сутки)  │
└───────────────────┘                 └─────────────────────────────┘
        │
        ▼ (полная история + LTV)
┌───────────────────┐    День 30+     ┌─────────────────────────────┐
│   СТУПЕНЬ 3       │ ──────────────► │  Сегментация по статусу     │
│  Полный профиль   │                 │  Удержание / добор / апсейл │
│  LTV + активность │                 │  Расчёт упущенной выгоды    │
└───────────────────┘                 └─────────────────────────────┘
```

## 3. Ступень 1 — Период знакомства (День 0–4)

### 3.1 Состояние данных

| Параметр | Значение |
|---|---|
| Доступные данные | Только регистрационные (ОКВЭД, возраст бизнеса, регион, тип, канал привлечения) |
| Транзакционная история | 0–3 операции (холодный старт) |
| Модели склонности | ❌ Не работают — нет истории |
| Решение | Статический мультиклассовый классификатор |

### 3.2 ML-модель: Мультиклассовый классификатор сегментов

**Задача:** Определить бизнес-портрет клиента в момент прихода в банк и сразу предложить релевантный якорный продукт.

**Алгоритм:** CatBoost Multiclass (возможные альтернативы — мета-ансамбль, MLP)

**Входные признаки (selected_features, 50 шт.):**

```python
# Регистрационные признаки

# Географические признаки
```

### 3.3 Инференс: Per-sample Feature Importance (SHAP)

На каждом инференсе модель возвращает не только предсказанный сегмент, но и **топ-5 признаков**, которые привели к этому решению. Это используется для:
- Передачи в LLM как контекст для генерации аргумента
- Объяснения решения в CRM-интерфейсе менеджера
- Логирования и аудита

```python
# Пример вывода инференса
{
  "predicted_class": "P1",
  "class_description": "🏪 Розничный продавец",
  "confidence": 0.829,
  "top5_feature_importance": [
    {"rank": 1, "feature": "okved_major_wrapped", "value": "retail_trade",
     "shap": +0.0101, "direction": "▲ в пользу сегмента"},
    {"rank": 2, "feature": "smb_type_code", "value": "2",
     "shap": +0.0050, "direction": "▲ в пользу сегмента"},
    {"rank": 3, "feature": "categ_name", "value": "grocery",
     "shap": +0.0038, "direction": "▲ в пользу сегмента"},
    ...
  ]
}
```

### 3.4 Генерация аргумента (Ступень 1)

**Режим:** Офлайн (batch). LLM используется **один раз при подготовке шаблонов** — не на инференсе в реальном времени. Это позволяет избежать latency и обеспечить контроль качества текстов.

```
Портрет P1 (Розничный ИП) × Продукт AME-7 (Эквайринг)
        │
        ▼
[LLM офлайн] → Шаблон аргумента с переменными:
"{{ИМЯ_ОТЧЕСТВО}}, для розничной точки карта Business One — 
 это не просто пластик. Вы получаете до 10% кэшбека 
 на категории, которые выберете сами..."
        │
        ▼
[Runtime] Подстановка: имя, отчество, город → готовый аргумент
```

**Итого шаблонов для Ступени 1:** 8 портретов × N якорных продуктов ≈ **24–32 уникальных шаблона**

## 4. Ступень 2 — Период активации (День 4–30)

### 4.1 Состояние данных

| Параметр | Значение |
|---|---|
| Доступные данные | Регистрационные + 30+ транзакций |
| Модели склонности | ✅ AUC > 0.65 — начинают работать |
| Профиль клиента | Видна регулярность, средний чек, типы контрагентов |
| Решение | Индивидуальные модели склонности + динамическая LLM-генерация |

### 4.2 ML-модели: Склонность к продуктам

Для каждого целевого продукта — **отдельная бинарная модель склонности** (CatBoost Binary Classification):

```
Модели Ступени 2:
├── model_propensity_acquiring.cbm    # Склонность к эквайрингу
├── model_propensity_zpp.cbm          # Склонность к зарплатному проекту
├── model_propensity_subscription.cbm # Склонность к подпискам
├── model_propensity_accounting.cbm   # Склонность к бухгалтерии
└── model_propensity_kopilka.cbm      # Склонность к налоговой копилке
```

**Ключевые поведенческие фичи (Ступень 2):**

```python
# Транзакционные агрегаты за 30 дней
sum_deb_fl_30          # Сумма переводов физлицам
cnt_deb_atm_30         # Количество снятий наличных
abm_paymentfiz_rate_1m # Доля платежей физлицам
balance_rur_amt_30_min # Минимальный остаток за месяц
avg_check_30           # Средний чек
cnt_contractors_30     # Количество уникальных контрагентов
has_salary_pattern     # Признак регулярных выплат физлицам
```

### 4.3 SHAP → DNA клиента → LLM

Алгоритм генерации аргумента на Ступени 2:

```
1. Рассчитываем score склонности к целевому продукту
        │
        ▼
2. Извлекаем топ-3 SHAP-признака для этого клиента
   (причины, почему модель считает его склонным)
        │
        ▼
3. Интерпретируем признаки в человекочитаемый текст:
   sum_deb_fl_30 = 450 000  →  "Вы активно переводите физлицам"
   cnt_deb_atm_30 = 12      →  "Вы регулярно снимаете наличные"
        │
        ▼
4. Формируем промпт для LLM:
   Клиент: {сегмент из Ступени 1} + {3 ключевых наблюдения} + {целевой продукт}
        │
        ▼
5. LLM #1 генерирует персонализированный аргумент
        │
        ▼
6. LLM #2 (compliance-checker) проверяет:
   ✗ Нет обещаний гарантированной прибыли
   ✗ Нет некорректных сравнений с конкурентами
   ✗ Нет запрещённых формулировок (ЦБ РФ, ФАС)
        │
        ▼
7. Валидный аргумент → CRM → доставка клиенту
```

**Режим работы:** Батч-генерация раз в сутки для всех клиентов в периоде День 4–30.

**Пример готового аргумента (Ступень 2, ЗПП):**
> «Иван Петрович, смотрю динамику по счёту — у вас серьёзный объём выплат сотрудникам или подрядчикам, почти полмиллиона в месяц. Сейчас вы тратите время и комиссию на эти переводы. Давайте оформим бесплатный зарплатный проект? Сотрудники получат карты с кэшбэком, а вы сможете зачислять им зарплату одной кнопкой без комиссий и очередей в банкомат.»

## 7. ML-компоненты и данные

### 7.1 Датасет для обучения классификатора (Ступень 1)

```
Файл: alfa_onboarding_dataset_5000.csv
Строк: 5 000 (сбалансированный, 625 строк × 8 классов)
Признаков: 50 (selected_features)
Таргет: P1–P8 (8 бизнес-портретов)
```

**Формирование разметки на реальных данных:**
1. Эвристическая разметка по правилам ОКВЭД + smb_type_code + days_from_ogrn
2. Верификация: якорный продукт портрета = первый подключённый продукт клиента
3. Конфликтующие примеры → ручная проверка аналитика

### 7.2 Параметры CatBoost-классификатора

```python
CatBoostClassifier(
    iterations           = 800,
    learning_rate        = 0.05,
    depth                = 7,
    l2_leaf_reg          = 3.0,
    loss_function        = "MultiClass",
    eval_metric          = "Accuracy",
    early_stopping_rounds= 80,
    auto_class_weights   = "Balanced",
)
```

**Результаты на синтетическом датасете:**
- Accuracy (val): **0.999**
- F1-macro (val): **0.999**
- CV 5-fold F1-macro: **0.999 ± 0.001**

> ⚠️ На реальных данных ожидаемые метрики: Accuracy 0.65–0.80, F1-macro 0.60–0.75 (из-за перекрытий между классами)

### 7.3 Per-sample SHAP (инференс)

```python
# Нативный CatBoost SHAP — без внешних библиотек
shap_vals = model.get_feature_importance(
    data           = pool,
    type           = "ShapValues",
    shap_calc_type = "Regular",
)
# shap_vals shape: (n_samples, n_features + 1, n_classes)
# Последний столбец — bias-терм, исключается из топ-5
```

Топ-5 признаков ранжируются по `|shap_value|` для предсказанного класса.

## 8. Генерация аргументов через LLM

### 8.1 Архитектура двойной LLM-проверки

```
┌─────────────────────────────────────────────────┐
│                 ARGUMENT ENGINE                  │
│                                                  │
│  [Портрет + SHAP-фичи + Продукт + Скрипт]       │
│              │                                   │
│              ▼                                   │
```

### 8.2 Режимы генерации по ступеням

| Ступень | Режим | Частота | Входные данные LLM |
|---|---|---|---|
| 1 | Офлайн (batch, один раз) | При запуске | Портрет + продукт + скрипт продажи |
| 2 | Онлайн-batch | 1 раз в сутки | SHAP-топ-3 + сегмент Ступени 1 + склонность |
| 3 | Онлайн-batch | По триггеру | Статус активации + LTV + история контактов |

### 10.2 Технологический стек

| Слой | Технология | Назначение |
|---|---|---|
| ML-моделирование | CatBoost 1.2+ | Классификатор сегментов + модели склонности |
| Feature Importance | CatBoost SHAP (нативный) | Per-sample объяснения на инференсе |
| Данные | Pandas, NumPy | Обработка и трансформация признаков |
| Валидация | Scikit-learn | Метрики, кросс-валидация |
| LLM-генерация | Anthropic Claude / OpenAI GPT-4o | Генерация аргументов + compliance |
| API | FastAPI | REST-эндпоинты для CRM-интеграции |
| Сериализация | CatBoost .cbm + Pickle | Сохранение моделей и метаданных |
